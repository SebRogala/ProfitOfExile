# ADR-006: gem_colors — Database-Backed UPSERT Lookup Table with In-Memory Resolver

## Status

Accepted

## Context

POE-19 requires mapping gem names to colors (RED/GREEN/BLUE/WHITE) to support the Font of Divine Skill strategy, which groups gems by color to compute win probabilities via the hypergeometric formula.

The existing color map lives in `/var/www/poe/data/gem-colors.json` (750 entries), generated by `fetch-gem-colors.mjs` from RePoE data. This file is accurate for the current league (3.28 Mirage) but must be updated each league when new gems are introduced or colors change.

Key considerations:

1. New leagues introduce new gems. The color resolver must be updatable without code changes or binary redeployment.
2. Transfigured gem names follow a pattern ("Cleave of Tenuity" → base "Cleave") but not all variants are in static data — the resolver must infer unknown colors via suffix stripping.
3. The resolution logic (Vaal prefix, Greater prefix, progressive " of X" suffix stripping) is non-trivial and should live in one place, not be duplicated between ingest and query paths.
4. The gem_colors map is read-heavy (once per collector run, resolved per gem per snapshot) and write-rare (new gems only).

## Decision

Store gem colors in a PostgreSQL table `gem_colors (name TEXT PRIMARY KEY, color TEXT NOT NULL CHECK (color IN ('RED', 'GREEN', 'BLUE', 'WHITE')))`.

A Go package `internal/price/gemcolor` provides a `Resolver` that:

1. Loads all rows from `gem_colors` into an in-memory map at startup.
2. Resolves unknown names via suffix-stripping strategies (Vaal prefix → Greater prefix → progressive ` of X` suffix).
3. Caches newly resolved names in memory during a run.
4. Writes new discoveries back to `gem_colors` via `INSERT ... ON CONFLICT DO NOTHING` at the end of each collector run, so future runs benefit from them without additional resolution. Existing entries are never overwritten — if a color correction is needed, it must be done via a direct SQL update or a new seed migration.

The initial dataset (750 entries from `gem-colors.json`) is seeded via a golang-migrate migration using `INSERT ... ON CONFLICT (name) DO UPDATE SET color = EXCLUDED.color` — idempotent and safe to re-run.

```go
// NewResolver loads gem_colors from DB into memory.
func NewResolver(ctx context.Context, pool *pgxpool.Pool) (*Resolver, error)

// Resolve returns the color for a gem name (direct + suffix-stripping).
func (r *Resolver) Resolve(name string) (color string, found bool)

// UpsertDiscoveries writes newly resolved names back to the DB.
func (r *Resolver) UpsertDiscoveries(ctx context.Context) error

// UnresolvedGems returns names that could not be resolved (for manual triage).
func (r *Resolver) UnresolvedGems() []string
```

## Consequences

### Positive

- League-to-league updates require only a SQL UPSERT — no code change, no redeploy.
- The resolver self-heals: suffix-stripping discovers new transfigured variants automatically, and `UpsertDiscoveries` persists them so they are resolved instantly next run.
- `UnresolvedGems()` surfaces gems that need manual color assignment, enabling a lightweight triage workflow.
- In-memory map keeps resolution at O(1) per gem despite DB-backed storage.

### Negative

- `NewResolver` requires a live DB connection at startup. If the DB is unavailable, the collector cannot start — no graceful degradation to a static fallback.
- Newly discovered colors from `UpsertDiscoveries` are visible to other processes only after write. Concurrent collector instances in the same run may duplicate resolution work (acceptable — UPSERT is idempotent).
- The seed migration embeds ~750 INSERT rows as a large SQL file. It is correct but not human-readable at a glance.

## Alternatives Considered

### Static JSON file bundled into the binary

Embed `gem-colors.json` via Go `embed` directive. Resolution runs entirely in-memory with no DB dependency.

**Rejected because**: Updates require a code change and binary rebuild for every new league. PoE leagues run every 3–4 months — this is a predictable maintenance burden that a DB-backed table eliminates entirely.

### Hardcoded map in Go source

Define gem colors as a `map[string]string` constant in the resolver package.

**Rejected because**: Identical to the static file approach but more invasive — a color correction requires editing Go source, not just a SQL statement. Also loses the audit trail that DB rows provide.

### Redis cache for color lookups

Store gem colors in Redis with a long TTL, populated from the existing JSON file.

**Rejected because**: Redis is available in the shared infra stack, but introducing a Redis dependency for a 750-entry lookup table is disproportionate. The data is stable (write-rare), relational (foreign-key-able), and already lives in PostgreSQL. A Redis-backed resolver adds operational complexity without performance benefit — the in-memory map is already O(1).

### Resolution at query time (no persistent storage)

Run suffix-stripping at query time, deriving colors on-the-fly from gem names without a lookup table.

**Rejected because**: The suffix-stripping algorithm requires knowledge of base gem colors to work (it strips suffixes until it finds a known base). Without the seed data, resolution degrades to guesswork. The lookup table is the foundation that makes the algorithm functional.
